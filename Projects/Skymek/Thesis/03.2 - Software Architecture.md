### 3.2.2.1 Atributos de Calidad de Software

Los atributos de calidad se definen según los escenarios de calidad propuestos por Bass et al. (2012), considerando los requerimientos técnicos del producto y las restricciones de software.

**Diagrama:** [TESIS-CALIDAD-atributos.wsd](../diagrams/TESIS-CALIDAD-atributos.wsd)

**Figura 3.4** - Atributos de Calidad del Sistema

**Tabla 3.1** - Escenarios de Atributos de Calidad

| Atributo | Prioridad | Estímulo | Respuesta | Medida | Táctica Arquitectónica |
|----------|-----------|----------|-----------|--------|------------------------|
| **Rendimiento** | Alta | Usuario solicita predicción | Sistema responde con predicción | < 500ms (sin cache), < 100ms (con cache) | Cache Redis, Connection Pool, índices DB |
| **Disponibilidad** | Alta | Falla de servicio | Sistema se recupera automáticamente | 99.5% uptime anual | Health checks, restart policies, réplicas |
| **Seguridad** | Alta | Intento de acceso no autorizado | Sistema bloquea y registra intento | 0 brechas de datos sensibles | JWT, Vault, TLS 1.3, RBAC |
| **Modificabilidad** | Media | Nuevo microservicio requerido | Desarrollador integra servicio | < 1 día de integración | Microservicios, API Gateway, contracts |
| **Testeabilidad** | Media | Cambio en código | Ejecutar suite de pruebas | > 80% cobertura de código | DI, Repository Pattern, mocks |
| **Usabilidad** | Media | Usuario nuevo se registra | Completar registro exitosamente | < 3 minutos, < 5 clicks | Mobile-first, UX intuitivo, validación inline |

**Restricciones de Software:**

| Restricción | Descripción | Justificación |
|-------------|-------------|---------------|
| Stack Open Source | Uso exclusivo de tecnologías open source | Reducción de costos de licenciamiento |
| Containerización | Todos los servicios en contenedores Docker | Portabilidad y escalabilidad |
| API REST | Comunicación síncrona vía REST/JSON | Simplicidad e interoperabilidad |
| Base de datos única | PostgreSQL como RDBMS principal | Consistencia y reducción de complejidad |

### 3.2.2.3 Modelo C4 - Nivel 3: Componentes

El diagrama de componentes muestra la estructura interna de los microservicios principales (Brown, 2023).

#### Componentes de Business Services (C4 Nivel 3)

**Diagrama:** [TESIS-C4-05-componentes-business.wsd](../diagrams/TESIS-C4-05-componentes-business.wsd)

**Figura 3.9** - Diagrama de Componentes Business Services

Los Business Services están compuestos por cuatro microservicios Java/Quarkus:

**ms-data-service (Gestión de Datos):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Controller | VehicleResource, TallerResource, MaintenanceResource | Endpoints REST CRUD |
| Service | VehicleService, TallerService, MaintenanceService | Lógica de negocio, validaciones |
| Repository | VehicleRepository, TallerRepository, MaintenanceRepository | Acceso a PostgreSQL via Panache |

**ms-workorder-service (Gestión de Citas):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Controller | AppointmentResource, ServiceOrderResource, MechanicResource | Endpoints REST |
| Service | AppointmentService, ServiceOrderService, MechanicService | Workflow de citas y órdenes |
| Repository | AppointmentRepository, ServiceOrderRepository | Acceso a datos con Panache |

**ms-auth-service (Autenticación):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Controller | AuthResource, UserResource | Login, registro, perfil |
| Service | AuthService, UserService | JWT, validación, roles |
| Client | VaultClient | Secrets management con HashiCorp Vault |
| Repository | UserRepository | Gestión de usuarios |

**ms-notification-service (Notificaciones):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Controller | NotificationResource, ScheduleResource | Envío y programación |
| Service | NotificationService | Selección de canal, templates |
| Channel | EmailSender, SmsSender, PushSender | Email, SMS (Twilio), Push (FCM) |

**Patrón de Arquitectura por Microservicio (Java/Quarkus):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Controller | REST Endpoints | Recibir peticiones HTTP, validación de entrada |
| Service | Business Logic | Implementar reglas de negocio, orquestación |
| Repository | Data Access | Acceso a PostgreSQL mediante Panache/JPA |
| DTO | Data Transfer | Mapeo entre entidades y respuestas API |

#### Componentes del ML Engine (C4 Nivel 3)

**Diagrama:** [TESIS-C4-04-componentes-ml.wsd](../diagrams/TESIS-C4-04-componentes-ml.wsd)

**Figura 3.10** - Diagrama de Componentes ML Engine

El ML Engine está compuesto por dos servicios Python que trabajan de forma coordinada:

**ms-prediction-service (Inferencia):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| API Layer | PredictionRouter, HealthScoreRouter | Endpoints REST para predicciones |
| Service Layer | PredictionService, HealthScoreService | Orquestación y cálculo de scores |
| ML Core | ModelLoader, FeatureBuilder, Predictor | Carga modelo, construye features, ejecuta inferencia |
| Infrastructure | CacheManager, DataClient | Cache Redis, comunicación con ms-data-service |

**ml-pipeline-xgb (Entrenamiento):**

| Capa | Componente | Responsabilidad |
|------|------------|-----------------|
| Pipeline API | TrainingRouter, EvaluationRouter | Endpoints para entrenar y evaluar |
| Orchestration | TrainingOrchestrator, SchedulerService | Orquesta pipeline end-to-end |
| Data Processing | DataExtractor, FeatureEngineering, DataSplitter | ETL y preparación de datos |
| Model Training | HyperparamOptimizer, ModelTrainer, ModelEvaluator | Optuna + XGBoost, métricas |

**Stack ML:**

| Librería | Versión | Uso |
|----------|---------|-----|
| XGBoost | 2.0.2 | Modelo de predicción (Gradient Boosting) |
| Optuna | 3.4 | Optimización de hiperparámetros (TPE) |
| scikit-learn | 1.3 | Preprocesamiento y métricas |
| pandas | 2.1 | Manipulación de datos |
| joblib | 1.3 | Serialización de modelos |

### 3.2.2.5 Evolución de la Arquitectura por Sprint

La arquitectura evoluciona incrementalmente en cada Sprint, siguiendo el principio de "just enough architecture" (Rubin, 2012).

**Tabla 3.3** - Evolución Arquitectónica por Sprint

| Sprint | Componentes Agregados/Modificados | Decisiones Arquitectónicas |
|--------|-----------------------------------|---------------------------|
| Sprint 1 | ms-auth-service, ms-data-service, ms-prediction-service, ml-pipeline-xgb | Definición de stack tecnológico, estructura de microservicios |

| Sprint 3 | SSE para notificaciones en tiempo real, WorkOrder model | Comunicación push, modelo de órdenes de trabajo |
| Sprint 4 | ms-notification-service completo, scheduler de alertas | Integración multicanal (Twilio, FCM, SMTP) |
